# 前端配合調整說明（資安修正：`/users` 權限與錯誤訊息）

> 對應後端修正：第 2 條（`GET /v1/users` 不再回密碼 + 加權限）、第 7 條（API 錯誤訊息改為通用訊息）。
> 此文件給前端：列出**會改變的 API 行為**與**前端需要配合的調整**。

---

## 1. `GET /v1/users` 行為變更（重點）

這支「列出整個學院使用者」的端點過去**任何登入者**都能呼叫、且**回傳每個人的明文密碼**。已收斂為：

| 項目 | 改版前 | 改版後 |
|---|---|---|
| 回應 `password` 欄位 | 回傳明文密碼 | **永遠為 `null`**（不再對外回傳密碼） |
| 誰能呼叫 | 任何持有有效 token 者 | **僅該學院的管理者(user_type 1)或教師(2)** |
| 可查的 `college_id` | 任意（可查別校） | **只能查自己所屬學校**（須等於登入身分的 college_id） |
| 權限不符的回應 | — | **HTTP 403**（含中文 `detail`） |

### 前端要做的事
1. **移除/隱藏使用者清單的「密碼」欄位**
   - 目前清單畫面會讀每筆使用者的 `.password` 來顯示，改版後該值恆為 `null`，欄位會空白。
   - 請直接拿掉這個欄位，或改成不顯示密碼（密碼本就不該顯示給任何人）。
   - 若有「查看/複製某使用者密碼」功能，請移除——後端已不提供此資料。
   - 註：`POST /v1/auth/login` 的回應 `user` **本來就不含 password**，這部分行為不變。

2. **只在「管理者／教師」的頁面呼叫 `GET /v1/users`**
   - 學生或一般使用者的畫面不要呼叫這支，否則會收到 403。
   - 呼叫時 `college_id` 請帶「登入者自己的 college_id」（即登入後拿到的身分學校），不要帶其他學校。

3. **處理 403**
   - 收到 403 時顯示後端回的 `detail`（中文），例如：
     - `僅該學院的管理者或教師可列出使用者`
     - `僅能查詢自己所屬學校（college_id）的使用者`
   - 不要把 403 當成「登入失效」去清 token / 強制登出（**401 才是 token 失效**）。

---

## 2. API 錯誤訊息（`detail`）改為通用訊息

過去伺服器內部錯誤（500）會把原始例外字串直接放在回應 `detail` 回給前端（會洩漏資料庫結構等內部資訊）。已全面改為：**內部錯誤只記在伺服器 log，回前端的是通用中文訊息**。

範例：

```jsonc
// 改版前（500）
{ "detail": "duplicate key value violates unique constraint \"Quiz_pkey\" ..." }

// 改版後（500）
{ "detail": "列出 Quiz 失敗，請稍後再試" }
```

### 前端要做的事
- 目前多處直接顯示 `response.data.detail`（約 27 處）——**這個用法可以維持不變**，`detail` 仍是可讀的中文訊息，照常顯示即可。
- ⚠️ **若有任何邏輯是用「比對 `detail` 的特定文字」來判斷錯誤類型**（例如 `if (detail.includes("xxx"))`），請改用 **HTTP status code** 判斷，因為 500 類訊息的文字已統一、不再帶細節。
- 4xx（如 400 參數錯誤、403 權限、404 找不到、401 未登入/過期）的 `detail` 仍是具體可讀訊息，可照常顯示。

---

## 3. 狀態碼對照（前端錯誤處理建議）

| Status | 意義 | 前端建議行為 |
|---|---|---|
| 401 | 未帶 token / token 無效或過期 | 清除本地 token、導回登入頁 |
| 403 | 已登入但**權限不足**（非管理者/教師、查別校等） | 顯示 `detail`，**不要登出** |
| 404 | 找不到資源 | 顯示 `detail` |
| 400 | 參數錯誤 | 顯示 `detail`（通常是表單驗證） |
| 500/503 | 伺服器/連線問題 | 顯示通用 `detail`，提示稍後再試 |

---

## 4. 前端驗收清單

- [ ] 使用者清單頁已移除/隱藏「密碼」欄位，畫面不因 `password=null` 出錯。
- [ ] 移除任何「查看/顯示/複製使用者密碼」的功能與按鈕。
- [ ] `GET /v1/users` 僅在管理者/教師頁呼叫，且帶自己的 `college_id`。
- [ ] 403 不會誤觸發登出流程（401 才登出）。
- [ ] 錯誤顯示仍正常（直接顯示 `detail`）；沒有依賴 500 訊息特定字串的判斷邏輯。

---

## 5. 不受影響（不用改）

- `POST /v1/auth/login`、`POST /v1/auth/refresh`：流程不變（login 回應本就不含密碼）。
- `GET /v1/users/me`：回自己的 profile，本來就不含密碼，行為不變。
- `PUT /v1/users/me/password`：改自己密碼，行為不變。
- 其他業務 API 的成功回應格式皆不變，僅「失敗時的 `detail` 文字」變通用。
